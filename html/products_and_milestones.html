Products and Milestones


Kitanda is a software stack made up of a number of services that are designed to work together but which can also be deployed in isolation. Our vision is for the entire stack to cover all the main business functions such as - but not limited to - Market Data, Pricing, Booking, Risk Management, Reporting and Confirmations. This page is the map on the road to that vision.

Products

Kitanda's vision for financial markets is embodied in two main projects - currently at different stages of development: Dogen, Kitanda's code generator, and Dirien, the distributed risk engine. Below you can find an overview of the current state of these products, their main milestones and the development roadmap. This page is updated at the end of every sprint.

You can click the application icons on the left to get more information about a product.

Dogen

Dogen is a code generator designed to target domain models. It is crucial for Kitanda because we use a intensively domain driven approach to software engineering, resulting in a very large number of domain specific languages. Dogen was created to make the modeling process simpler: the user creates a domain model using a UML tool and Dogen uses it to generate its source code representation. The generated code contains most of the services required from a typical C++ domain object such as serialisation, hashing, streaming and so on.

The main objective of creating a domain generator is to avoid having to maintain manually a significant amount of trivial code; this not only speeds up the development process but it also improves quality as programmers do not tend to perform repetitive tasks terribly well. We developed our own domain generator because we could not find one that fitted our requirements - open source or otherwise.

Note that Dogen is specifically tailored for our needs and as such it is not a general purpose domain generator. We are, however, wiling to accept any patches for functionality not directly required by us.

Architecture

Component Description
Front-end Reads documents generated by a UML graphical tool. At present we are only targeting Dia, the Gnome diagram tool, because its simple enough for our needs. However, the front-end is designed to be extended to support other formats such as XMI.
Middle-end Intermediate representation of the domain model, designed with domain generation in mind. We created the Simplified Modeling Language (SML) for this, a bare-bones meta-model inspired by EMF's ECore.
Back-end Outputs a source code representation of the domain model in any of the supported programming languages (grammars in Dogen speak). At present we only support C++-11 but SQL is on our road map.

Milestones

Setup the development infrastructure
Setup the git repositories at GitHub. Setup Continuous Integration (CI) on CDash across all the supported platforms - including build machines, build agents, scripts, etc. Finally, create the agile bookkeeping infrastructure to manage sprints, stories, etc.
Create the foundations for reading Dia diagrams
Create the code infrastructure required to parse the XML produced by Dia and to convert it into our internal representation of their model. Add transformation code that can translate the Dia model into SML.
Create the foundations for C++-11 code generation
Add support for C++-11 code generation, including the main facets: serialisation, hashing and streaming. Create a command line driver with required options. At this stage Dogen establishes a full pipeline from drawing to valid code generation - a significant advancement in the architecture. It is producing compilable and testable code.
Add support for single inheritance
Inheritance relationships described in UML diagrams need to be carried through to C++-11. Support for multiple inheritance is not be part of this task - it will only be implemented when we find a domain model that requires it. We are currently in the analysis phase for this feature.
Add support for composition
Composition relationships described in UML diagrams need to be carried through to C++-11. Once this task is implemented Dogen will be able to express all the structural features required for Kitanda. We are currently in the analysis phase for this feature.
Add initial support for the STL model
Once implemented, Dogen will be able to correctly interpret the usage of most common STL classes used in domain modeling such as std::string, STL containers, smart pointers, etc. Work has not yet started for this feature.
Add initial support for the Boost model
Once implemented, Dogen will be able to correctly interpret the usage of most common Boost classes used in domain modeling such as Boost containers, smart pointers, etc. Work has not yet started for this feature.
Support for SQL generation
Dogen will support the creation of special-purpose domain models that have CRUD support, allowing the persistence of domain objects on a relational database. The generated code is the SQL required to create and drop tables, schemas, etc - it is the relational representation of the domain. We only intend to support Postgres. The design work has been carried out for this feature, and initial implementation has started.

Dirien

Dirien - the distributed risk engine - is designed to augment quantitative analysis libraries such as Quantlib with all the required facilities needed by production systems: portfolio support, result aggregation, distributed processing (map/reduce), scenario analysis and much more.

Architecture

Component Description
Risk domain model Definition of the domain language required to model risk, with entities such as instruments, portfolios, term structures, results (Greeks), etc. The risk domain model is based on the domain model created by Quantlib but extended with high-level constructs that are not needed at the pricing engine level.
Quant engine wrapper API Exposes a common API for Dirien to access the pricing engines provided by different quantitative libraries. Also includes a domain model that defines the available configuration for each wrapped library (e.g. supported models, instruments and Greeks for each model, etc). Initially we will provide support for Quantlib.
Valuator The core of the risk engine, the Valuator makes use of the risk domain model and the Quant engine wrapper API to calculate results for sets of instruments. The Valuator works on a single dimension of risk.
Risk space The risk space domain model augments the risk model with support for multi-dimensional scenario analysis. It defines the risk space - an hyperspace in which objects of the risk domain model exist; a language to define sparse representations of the risk space; and containers that instantiate those representations. The library also provides the transformations required to move risk domain model objects in risk space.
Distributed and Map/Reduce This is the service tier of Dirien, with the traditional client/server separation: the client is responsible for submitting requests for valuation and the server is responsible for producing the valuation results. Internally the server makes use of a map/reduce approach: jobs are split into multiple batches according to batching algorithms; workers in the cloud perform the number crunching - each of which running a Valuator; finally the results are aggregated and sent back to the client. The batching takes into account multi-dimensional scenarios, via the risk space library.

Milestones

Create the foundations for the risk domain model
Initially we want to create a minimal domain model required to value a simple set of instruments such as vanilla options. This domain model includes entities such as term structures and instruments. The design work has started for this feature.
Create the foundations for the Valuer and wrappers
The objective is to define a configuration model for QuantLib, exposing a small number of pricing engines. We then need to implement functionality in both the Valuer and the wrapper API to value a set of instruments. The design work has started for this feature.
Create the client/server foundations
Define a protocol that enables clients to send requests for valuations to the server; wrap the valuation functionality thus far created in a server which is capable of returning results. This feature has not been started yet.
